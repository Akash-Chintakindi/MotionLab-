## Project Overview

**MotionLab** is a Brilliant-inspired interactive web app for teaching AP Physics C kinematics to high school students. The core philosophy: learn by doing — students are dropped into problems, graphs, and simulations before any explanation. The 7 lessons (Learn/Practice/Quiz) are hand-authored; the **AI Practice Hub** (`/practice`) is the one AI-powered surface, generating adaptive problems via Firebase AI Logic (Gemini).

- **Tech Stack:** React 18 + Vite 8 + TypeScript + Tailwind CSS v3
- **Backend:** Firebase Auth (email/password) + Cloud Firestore
- **Dev Environment:** Firebase Emulator Suite with `demo-motionlab` project ID; portable Temurin JDK 21 (arm64) stored in `.tools/` (gitignored), JAVA_HOME wired into npm scripts

- **Routing:** React Router v6; `react-router-dom`
- **Firebase project:** `motionlab-3e44a` — **deployed live at https://motionlab-3e44a.web.app** (`npm run deploy`)
- **AI:** Firebase AI Logic (Gemini Developer API, free tier) via `firebase/ai`. Requires one-time provisioning: `npx -y firebase-tools@latest init ailogic` (enables the Gemini Developer API). App Check is NOT yet configured — REQUIRED follow-up before heavy production use (protects the AI quota).
- **Fonts:** Space Grotesk (display) + Inter (body), loaded via Google Fonts

## Course Structure: AP Physics C Kinematics (7 lessons)

> App-facing course title is **"AP Physics C Kinematics"** (the "Motion Through Calculus" suffix was removed at the user's request).

### Lesson 1: Position, Velocity, and Slope
- **Core idea:** Velocity = derivative of position; v = dx/dt
- **Covers:** Position-time graphs, slope as average velocity, tangent slope as instantaneous velocity, positive/negative/zero velocity, v(t) = dx/dt
- **Interactive elements:** Drag point along x(t) curve; live tangent line; velocity readout; secant/instantaneous mode; region prediction (positive/negative/zero velocity); velocity sign sort
- **Lesson 1 detailed 10-step flow:**
  1. Hook: "A cart moves along a track. Where is it moving fastest?"
  2. Explore: drag a point on the x(t) curve; tangent appears live
  3. Predict: tap the steepest region (wrong answer feedback: "Look for steepest slope, not highest position")
  4. Average velocity: drag two time markers; see secant slope
  5. Instantaneous velocity: markers shrink; tangent limit appears
  6. Derivative reveal: introduces v(t) = dx/dt
  7. Concept screen: derivative notation
  8. Sort: classify labeled points as positive/zero/negative velocity
  9. Application: predict motion from a new position-time graph
  10. Completion: mastery result, streak update, Lesson 2 unlock

### Lesson 2: Velocity, Acceleration, and Changing Motion
- **Core idea:** Acceleration = derivative of velocity; a(t) = dv/dt
- **Covers:** v(t) graphs, slope as acceleration, speeding up vs. slowing down
- **Interactive:** Adjust velocity graph segments; motion animation responds; identify when object speeds up/slows down

### Lesson 3: Displacement from Area Under Velocity
- **Core idea:** Displacement = integral of velocity; Δx = ∫v(t)dt
- **Covers:** Area under v(t) graph, positive/negative area, net displacement vs. total distance
- **Interactive:** Shade graph regions; drag interval endpoints; compare net displacement and total distance

### Lesson 4: From Acceleration to Velocity and Position
- **Core idea:** Integration builds position from velocity and velocity from acceleration
- **Covers:** v(t) = v₀ + ∫a(t)dt; x(t) = x₀ + ∫v(t)dt; constant acceleration equations; initial conditions
- **Interactive:** Slider for constant acceleration; live a(t)/v(t)/x(t) graph updates; predict final velocity and displacement

### Lesson 5: Motion in Two Dimensions
- **Core idea:** 2D motion = two linked 1D motions
- **Covers:** Position vector r(t); velocity vector v(t) = dr/dt; acceleration vector; components and vector arrows
- **Interactive:** Move object on 2D plane; display x/y position, velocity, and acceleration components; match vector arrows to motion
- **Note:** All subscript notation (v_x, v_y, a_x, a_y) uses the `RichText` renderer for proper `<sub>` tags

### Lesson 6: Projectile Motion
- **Core idea:** Horizontal and vertical motion are independent
- **Covers:** Constant horizontal velocity; vertical acceleration due to gravity; parametric equations; time of flight, peak height, range
- **Interactive:** Launch angle + speed sliders; projectile path updates continuously/live (no launch button); ball animates in loop; stats (peak, range, time of flight) shown live

### Lesson 7: Kinematics Mastery Challenge
- **Core idea:** Combine graphs, equations, derivatives, integrals, and 2D reasoning
- **Covers:** Mixed graph interpretation; translating between x(t), v(t), a(t); AP-style conceptual problems; multi-step reasoning
- **Interactive:** Build missing graph from a given graph; multi-step motion scenario; review weak areas

## Content Model (TypeScript, version-controlled — never in Firestore)

```ts
Course {
  id: string
  title: string
  lessons: Lesson[]
}

Lesson {
  id: string
  title: string
  order: number
  estimatedMinutes: number
  steps: LessonStep[]
}

LessonStep {
  id: string
  type: "concept" | "multipleChoice" | "graphDrag" | "sliderSimulation" | "sort" | "numeric"
  prompt: string
  interactionConfig: object  // curve key, options, tolerance, etc.
  correctAnswer: object
  feedback: {
    correct: string
    incorrect: string
    hint?: string
  }

```

- Lesson content is canonical in code; Firestore stores **only user progress**
- Stable deterministic IDs required: e.g. `lesson-1-position-velocity`
- Curve registry (`src/lib/curves.ts`): step configs reference curves by string key (e.g. `"parabola_down"`, `"smooth_increasing"`)
- `RichText` component (`src/components/RichText.tsx`): parses `v_x`, `v_y`, `a_y` → `<sub>` tags; applied to all prompts and feedback text


### Data Flow
1. User signs in via Firebase Auth
2. React loads local course content from TypeScript files
3. App reads Firestore progress for authenticated user
4. Lesson renderer combines local content + user progress
5. On every step completion, React writes progress to Firestore
6. On lesson completion: Firestore updates completedLessonIds, updates streak + BEST mastery (does NOT unlock the next lesson — that requires passing the quiz at ≥70%; see "Mastery + quiz gating")
7. On return: app restores user to saved lesson and step

## Key Architectural Decisions

- **Content canonical in code; only progress in Firestore** — keeps lesson definitions version-controlled, fast to render, AI-generatable later
- **Completion badge based on `completedLessonIds`** — never removed when restarting a lesson; replays don't break the "Completed" indicator
- **Restart clears lessonProgress doc** (resets step tracking) but does NOT modify `completedLessonIds` or unlock logic
- **Lesson engine phases:** `play` → `summary` → `review`; completed lessons open to summary screen instead of auto-restarting
- **`answered` Set replaces `locked` boolean** — tracks which steps have been correctly answered; enables back-navigation with prefilled correct answers
- **MCQ requires Submit button** — selecting an option doesn't auto-grade; user must click Submit (re-submit after wrong answer); prevents spam-clicking
- **Back button in lesson** — "← Previous step" button visible at step > 0; revisiting shows the step in its solved, locked state with correct answer prefilled
- **Lazy route loading** — all page components are `React.lazy()`; Firebase and vendor chunks split manually for <2s first load

## Lesson Modes (Learn / Practice / Quiz)

Each dashboard lesson card is an **expandable dropdown** (`src/components/LessonCard.tsx`) with three modes. Locked lessons are not expandable. The card surfaces best Quiz % and best Practice % per lesson.

### Routes
- `/lesson/:id` — Learn (existing lesson engine)
- `/lesson/:id/practice` — Practice game (`src/pages/PracticePage.tsx`)
- `/lesson/:id/quiz` — Quiz (`src/pages/QuizPage.tsx`)

### Mastery + quiz gating (IMPORTANT — see "Learning Science.txt")
Two performance thresholds, centralized in `src/lib/gating.ts`:
- `LESSON_MASTERY_THRESHOLD = 0.8` and `QUIZ_PASS_PERCENT = 70`.
- Helpers: `modesUnlocked(courseProgress, lessonId)` (best mastery ≥ 80%), `quizPassed(scorePct)` (≥ 70%), `lessonMastery(...)`.

Rules:
1. **Practice + Quiz of a topic unlock only after Learn is mastered at ≥ 80%.** Below 80%, both modes stay locked on the `LessonCard` (note "🔒 Master Learn (80%)") and their routes (`PracticePage`/`QuizPage`) show a "Master the lesson first" guard. The lesson completion screen branches: ≥80% → 🎉 + next CTA; <80% → 📊 "Almost there!" with **Review answers** (`onReview`) and **Redo lesson** (`onRestart`), no forward CTA.
2. **The next topic unlocks only when the quiz is passed at ≥ 70%.** `QuizPage.onComplete` always records the score but calls `unlockNextLesson` **only if `quizPassed(scorePct)`**. Below 70%, `QuizRunner`'s score screen hides the next-step CTA, shows a retake notice (`data-testid="quiz-retry-notice"`), and the retry button reads "Retake quiz". `QuizRunner` takes a `passThreshold` prop (QuizPage passes `QUIZ_PASS_PERCENT`; default 0 = no gate).
- `completeLesson` stores the learner's **BEST** mastery (max of prev and new), so a topic stays unlocked (and accuracy badges stick) even after a weaker replay. `unlockNextLesson(uid, lessonId)` still does the actual unlock.
- `useLessonEngine` exposes `completion.practiceUnlocked` (best mastery ≥ 80%) and now caches the per-step summary on finish so "Review answers" works straight from the completion screen; `onReview` clears `completion` so review steps render.

### Completion chaining: Learn → Practice → Quiz → next lesson
- `src/lib/lessonFlow.ts` `nextDestination(lessonId, mode)` computes the next CTA, gracefully **skipping modes a lesson doesn't have** (uses `hasGame`/`hasQuiz`). learn→practice→quiz→next-lesson; missing modes are skipped; last lesson with nothing after → null (Back to course only). NOTE: the forward CTA is gated by the thresholds above (completion screen only shows it at ≥80% mastery; quiz score screen only shows it at ≥70%).
- Every "next" CTA (Learn completion screen, game result screen, quiz result screen) uses `data-testid="next-step"`.
- The 🎉 celebration completion screen and all "Back to course" links are preserved.

### e2e note (gating)
- `e2e/helpers.ts` `passQuiz(page, lessonId)` answers a quiz **correctly** by importing `getQuiz` from `src/content/quizzes` (quiz modules import only types, so this is safe in Playwright). Progression specs (lesson1, lesson-sequence, full-course) use `passQuiz` to clear 70%. `finishQuiz` (arbitrary answers) is kept for fail-path use. quiz1/practice1 specs first `completeLesson1` (90–100% mastery) so the quiz/practice are unlocked.

### Practice games — registry + contract
- Registered in `src/games/registry.ts` keyed by lessonId via `lazy(() => import("./XGame").then(m => ({ default: m.XGame })))`. Helpers `getGame(lessonId)` / `hasGame(lessonId)`.
- **GameProps** (registry.ts): `{ onScore?: (bestOverall: number) => void; next?: { href: string; label: string } }`.
- A game MUST: be self-contained & presentational (no Firebase imports); call `onScore(best, 0–100)` exactly once when the player finishes; render a result screen with a primary `data-testid="next-step"` react-router `Link` when `next` is provided, plus a "Play again" control and a "Back to course" `Link` to `/`; respect `usePrefersReducedMotion` (skip animation, no infinite rAF) ; be mobile-first; use `requestAnimationFrame` for animation (not CSS transitions) for 60fps.
- Persistence handled by `PracticePage` (`recordPracticeScore` keeps best; counts as daily streak activity). Best score appears on the dashboard dropdown.
- Pure scoring lives in `src/games/lib/*Scoring.ts` with Vitest unit tests.
- **Reference implementation:** `src/games/DriveTheCartGame.tsx` + `src/games/lib/driveScoring.ts` (Lesson 1).

### Drive the Cart (Lesson 1) — current design
- Set a per-second velocity profile (sliders, t=0..6, dt=1) to reproduce a target position-time path across 3 rounds (constant → stop-and-hold → over-the-hill).
- **Difficulty design (important pattern):** NO full target line is drawn — only a few **checkpoint rings** (at t=3 and t=6) are shown, and they turn green when the path passes through them. Scoring still grades the WHOLE curve (RMS error → 0–100), so sparse anchors make it harder, not just leniently scored. New games should follow this "few visible anchors, full-curve scoring" idea where applicable.
- Live score; Play animates the cart (skipped under reduced motion); overall best = average of per-round best scores.

### Quizzes — registry + contract
- Registered in `src/content/quizzes/index.ts`: `getQuiz(lessonId)` / `hasQuiz(lessonId)`.
- **Types** (`src/types/content.ts`): `QuizQuestion { id; type: "multipleChoice"|"numeric"; category: "conceptual"|"calculation"; prompt; graph?; plot?; options?; correctOptionId?; value?; tolerance?; unit?; placeholder?; explanation }`; `Quiz { lessonId; questions }`.
- 8–10 ORIGINAL questions per lesson, mixing conceptual + calculation; every question has a non-empty `explanation`. Must be fully answerable from that lesson's Learn content. May reuse `graph` (GraphConfig) / `plot` (PlotConfig) visuals.
- Engine: `src/hooks/useQuiz.ts` + presentational `src/components/quiz/QuizRunner.tsx` (immediate per-question feedback → final score + per-question review with retry). `QuizPage` wires persistence (`recordQuizScore` best; daily streak) + the `next` CTA.
- Pure grading: `src/lib/quiz.ts` (`gradeQuestion`, `quizScorePct`). **Reference:** `src/content/quizzes/quiz-1-position-velocity.ts` + `src/content/quizzes/__tests__/` / `src/content/__tests__/quiz-content.test.ts`.

### Practice games by lesson (names + concept)
- L1 **Drive the Cart** — build a velocity profile to hit position checkpoints (DONE)
- L2 **Acceleration Pilot** — set an acceleration profile to match a target velocity curve (a = dv/dt)
- L3 **Area Painter** — choose/shade the area under v(t) to hit a target displacement (signed area; net vs total)
- L4 **Integral Stack** — integrate a(t) → v(t) → x(t); tune acceleration (+ v₀/x₀) to hit position targets
- L5 **Coordinate Courier** — set v_x, v_y (and/or a_x, a_y) to route to 2D waypoints (independent components)
- L6 **Cannon Range** — pick launch angle + speed to hit targets at varied range/height (projectile independence)
- L7 **Boss Rush** — a fast scored gauntlet of mixed mini-challenges across all prior topics

## AI Practice Hub (`/practice`) — adaptive AI problems

A separate surface from the per-lesson Practice **games** (`/lesson/:id/practice`). Student picks a **topic** (any of the 7 lessons) + **difficulty** (easy/medium/hard); an AI generates one problem at a time with immediate feedback (graded client-side like a lesson step). Adaptive nudges: a trailing run of correct answers suggests leveling up (unless on hard); a run of wrong answers suggests leveling down, or — when already on easy — going back to the lesson to review.

- **Shared contract (`src/ai/`):**
  - `practiceTypes.ts` — `Difficulty`, `DIFFICULTIES`, `AIPracticeQuestion`, `PracticeTopic`, `GenerateParams` (dependency-free).
  - `topics.ts` — `practiceTopics()` derives topics from `course.lessons` (id, title, blurb=coreIdea).
  - `adaptive.ts` — pure `adaptiveSuggestion(results, difficulty)` (+ `harderDifficulty`/`easierDifficulty`, `ADAPTIVE_STREAK=3`). Unit-tested in `src/ai/__tests__/adaptive.test.ts`.

## Games Hub (`/games`) + arcade games

`src/pages/GamesPage.tsx` — a picker (`data-testid="games-page"`) linking to two full arcade games (cards `game-card-pool`, `game-card-basketball`). These are SEPARATE from the per-lesson practice games on lesson cards.

### Shared arcade foundation (`src/games/arcade/`)
- `audio/audioEngine.ts` — **procedural Web Audio** (no audio files): `resumeAudio()` (call from a user gesture), `setMuted`/`isMuted`, `playSfx(name)`, `startMusic(track)`/`stopMusic()`, and two distinct loops `POOL_TRACK` (cool lounge groove) / `BASKETBALL_TRACK` (energetic). `audio/useArcadeAudio.ts` React wrapper + `MuteButton.tsx`.
- `useGameLoop.ts` — rAF loop hook `useGameLoop(cb(dt), running)` with clamped dt.
- `useHighScore.ts` + `services/gameScoreService.ts` — best score per gameId in Firestore (`users/{uid}/gameScores/arcade`) with a localStorage fallback (works logged-out/offline). NOTE: it does NOT auto-post to the public leaderboard anymore — that's an explicit opt-in from the game-over screen (see Leaderboard below).
- `types.ts` — `ArcadeGameProps { highScore; onGameOver({score}); leaderboard? }`. The optional `leaderboard: ArcadeLeaderboard { playerName; playerId; publish(score); fetchTop(topN?) }` lets a game OFFER to post the score + show the board without importing Firebase (the page injects the Firestore-backed callbacks). Pages (`PoolGamePage`/`BasketballGamePage`) wrap the game in `AppShell` and persist the score via `useHighScore`. Games themselves NEVER import Firebase.
- Both games are React + **HTML Canvas with custom deterministic physics** (NOT Phaser), built per the `game-designer` + `physics-intuition` skills (juice, readable-at-thumbnail, real arcs/weight), and respect `usePrefersReducedMotion`.

### Physics Pool (`/games/pool`)
- Authentic cue→object-ball→pocket on a coordinate-plane table. The shot is CALCULATED and entered numerically (angle° + speed); HUD shows cue/object/target-pocket coords, ball radius, and friction `a`; a help panel teaches the **ghost-ball** aim and **v=√(2·a·d)**. After each shot it reveals the ideal angle/speed + the player's angle error.
- Pure module `pool/poolPhysics.ts` (deterministic sim with sub-stepping, equal-mass elastic collisions, friction, cushions, pockets; `computeIdealShot` ghost-ball; `gradeShot`) — unit-tested incl. an integration test proving the ideal shot actually sinks the ball. Component: `pool/PoolGame.tsx` (+ `poolLevels.ts`, `poolRender.ts`, `PoolHud.tsx`). Root `data-testid="pool-game"`.

### Buzzer Beater basketball (`/games/basketball`)
- **FRONT-FACING redesign** (GamePigeon-style + Madden-kick aiming). You look straight at a hoop mounted on a brick arena wall: backboard + rim centered up top, net hanging below, the shot **clock floats ABOVE the hoop**, and the ball spawns on a **shooting line near the bottom at a RANDOM x** each shot.
- Pick a clock (30/60/120s). Per shot, TWO timed inputs (Space/tap/click; button reads "LOCK AIM" → "SHOOT!"):
  1. **Aim** — a glowing dotted **parabola (the aim arc)** sweeps side to side at a FIXED `trueSpeed` (the speed that scores from the dead-on direction), so the launch DIRECTION is the skill: you time the lock when the arc drops through the hoop. Its angle range ADAPTS to the ball's spawn x (`aimRange`): a centered ball sweeps around straight-up; a far-left ball sweeps through up-and-right directions (mirror for far-right).
  2. **Power** — a vertical meter whose **green band is the true-aim MAKE window**, computed by sampling speeds with `solveSpeedWindow(centerRad)` and keeping the contiguous range that actually goes in. The band is fixed at shot start (does NOT depend on the locked direction), so "stop it in the green" genuinely corresponds to a make WHEN the aim is true. **The aim arc does NOT oscillate during the power phase** — it freezes at lock (locked dir + `trueSpeed`) and only the meter moves; the arc is green/red per `aimGood = simulateShot(lockedDir, trueSpeed).made`.
- **Real projectile physics + rim collision + RATTLE** (`simulateShot`, deterministic, no rng): the ball integrates under gravity and each descending rim-plane crossing is classified — center ⇒ **swish**, within the ring ⇒ **rim-in**, far side ⇒ **bank in**. A graze in the clip zone (rimRx…rimRx+ballR) **bounces** (funnels toward center when near the hole, kicks outward at the edge, damped each contact, up to 2 contacts) and may drop in on a later descent or finally clank out — so a slightly-off aim with good power rattles around and OCCASIONALLY goes in (closer aim ⇒ more likely). Outcome drives banners, particles, net swish (`netStretch`), and audio.
- Every 15s (game clock PAUSES) a question modal appears: pick easy/medium/hard → a question from the practice bank → correct adds time (+2/+5/+10), wrong −5s. Clock to 0 ends it; tracks high score.
- **Module split (frozen contracts so physics + renderer stay in sync):**
  - `basketball/basketballLayout.ts` — geometry source of truth: `ASPECT_W/H`, `Layout`, `computeLayout`, `randomSpawnX`. Hoop/rim/board/spawn-line positions, plus `gravity`/`speedMin`/`speedMax` (all scale with canvas height for resolution independence).
  - `basketball/basketballScene.ts` — the render contract: `BballScene` (incl. `aimPreview`, `aimLocked`/`aimGood`, `power`/`powerGreen`/`powerHalf`, `netSway`/`netStretch`, `ballScale`), `Phase`, `Screen`, `Particle`/`FloatText`/`CrowdSpeck`/`Banner`.
  - `basketball/basketballPhysics.ts` — pure math (`aimRange`, `aimAngleAt`, `launchVelocity`, `idealSpeed`, `powerToSpeed`/`speedToPower`, `solveSpeedWindow` [the green make-window], `greenBand`/`aimGood` [legacy helpers, still exported], `simulateShot` → `ShotResult`/`ShotSim` [now with up-to-2 rim rattle bounces], `previewArc`, `applyTimeDelta`, pacing helpers, consts `TIME_BONUS`/`WRONG_PENALTY`/`TIME_LIMITS`/`QUESTION_INTERVAL`) — **angles are radians from +x in a y-UP sense (π/2 = straight up); screen y grows downward.** Unit-tested (31 tests: adaptive aim, true-aim make-window scoring across spawns, rim-rattle spectrum swish→rim-in→bounce-out, short/long misses).
  - `basketball/basketballRender.ts` — pure canvas painter `drawScene(ctx, layout, scene)` (front-facing arena/crowd/spotlight, backboard, two-piece rim with net hanging inside, net swish, ball + trail, the sweeping aim arc, dynamic power meter, clock-above-hoop HUD, particles/banner/flash). Built with the `/game-designer` + `/physics-intuition` skills.
  - Component `basketball/BasketballGame.tsx` (+ `QuestionModal.tsx`) owns simulation/state; mirrors `phase` into `uiPhase` state for the button label. Root `data-testid="basketball-game"`, controls `bball-start`/`bball-action`.
  - **Sizing**: the court is no longer pinned at 420px — its wrapper uses `style={{maxWidth:"min(36rem, calc((100svh - 15rem) * 0.8))"}}` so it grows to fill the screen on desktop yet stays capped by viewport HEIGHT (it's a tall 4:5 portrait court) and goes full-width on phones. The end `Overlay` is `overflow-y-auto` so the richer game-over content never clips.
  - `basketball/EndLeaderboard.tsx` — game-over leaderboard **opt-in** (pure UI; uses the injected `leaderboard` callbacks, no Firebase). Flow: ask ("Add your score? Posting as {name}") → on "Add my score" (`bball-add-leaderboard`) it `publish`es then `fetchTop(8)` and shows a mini board (`bball-end-leaderboard`) with the player's row highlighted alongside other users; "No thanks" dismisses; errors offer retry. Unit-tested `basketball/__tests__/EndLeaderboard.test.tsx` (4 tests).

### Cross-user Leaderboard (`/leaderboard`)
- Nav tab "Leaderboard" (`AppShell`, after Games) → `src/pages/LeaderboardPage.tsx` (default export, behind `RequireAuth`). A game picker (`lb-game-basketball`/`lb-game-pool`, default basketball) + a top-20 list (`data-testid="leaderboard-page"`/`leaderboard-list`, rows `lb-row-{uid}`), medal-tinted ranks, current user's row highlighted with a "You" badge, loading + empty states.
- **Separate PUBLIC store** (the per-user `users/{uid}/gameScores/arcade` doc is private-by-rules, so it can't power a cross-user board). `src/services/leaderboardService.ts`: model `leaderboards/{gameId}/entries/{uid}` (doc id = uid) shaped `{ uid, name, score, updatedAt }`. API: `recordLeaderboardScore(uid, name, gameId, score, fdb?)` (upsert, only RAISES the score, name capped 24 chars / "Anonymous"), `getLeaderboard(gameId, topN=20, fdb?)` (subcollection `orderBy(score desc) limit` — single-field auto index, NO composite index needed), `LeaderboardEntry`, `LEADERBOARD_GAMES`.
- **Posting is OPT-IN** (not silent): after a basketball game ends, `EndLeaderboard` asks the user; only on confirm does it call `recordLeaderboardScore(uid, name, "basketball", score)` and then show the board. `BasketballGamePage` builds the `leaderboard` prop from `useAuth` + `leaderboardService` (`name = displayName || email-prefix || "Anonymous"`). (Pool doesn't wire the opt-in yet — the mechanism is generic, so add the same `leaderboard` prop to `PoolGamePage` to enable it there.)
- `firestore.rules`: signed-in users may READ any leaderboard entry; a user may create/update ONLY their own entry (`uid == auth.uid`, validates score:number≥0, name:string), never delete. **⚠️ MUST be deployed for the live app to work** — the rules are written but NOT yet deployed (the live leaderboard reads/writes are denied until then). Deploy requires re-auth: `firebase login --reauth` then `npm run deploy:rules` (the Firebase CLI token is currently expired). Works against the local emulator already (it reads the rules file directly). Unit test `src/pages/__tests__/LeaderboardPage.test.tsx`; emulator integration test `src/services/__tests__/leaderboardService.integration.test.ts`.

## Key Decisions & Constraints to Preserve

1. **Lesson content stays in TypeScript files** — never move to Firestore
2. **`completedLessonIds` is the permanent completion source of truth** — restart does not remove from this list
3. **AI is confined to the Practice Hub** — lesson/quiz/game content stays hand-authored; only `/practice` calls a model (Firebase AI Logic / Gemini). Set up App Check before relying on it in production.
4. **MCQ requires Submit** — do not revert to instant-grade-on-selection
5. **Back button restores solved state** — prefillCorrect must work for all graded step types
6. **60 FPS on graphs/sims** — use requestAnimationFrame, not CSS transitions
7. **All content must have stable IDs** — never change existing step/lesson IDs (breaks progress data)
8. **Firestore security rules** — all progress writes scoped to authenticated `userId`
9. **Test-driven** — every new feature needs unit tests + e2e coverage before marking done
